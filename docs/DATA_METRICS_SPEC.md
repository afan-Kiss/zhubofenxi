# 数据指标口径规范

本文档定义经营 BI 看板的核心统计口径。任何改动金额、退款、品退、订单数、买家标识的代码，**必须**对照本文档并更新验收说明。

**本系统所有经营日期按中国时间 Asia/Shanghai 统计。** 自定义日期 `startDate=YYYY-MM-DD` 解析为当日 00:00:00.000 ~ 23:59:59.999（Asia/Shanghai），不依赖服务器本地时区。

---

## 1. 支付金额

| 项 | 口径 |
|----|------|
| 时间字段 | 按订单 **支付时间** 统计 |
| 计入条件 | 有支付时间且状态为 **已支付** |
| 取消/退款 | 已支付后取消或退款，**仍计入**支付金额 |
| 退款冲减 | 退款 **不冲减** 支付金额 |
| 排除 | 未支付、无支付时间的订单 |
| 线下成交 | 见 **§1.1**；计入同一套「支付金额 / 总 GMV」 |

**验收方法**：选定日期范围，逐单核对 `payTime` 与支付状态；支付金额 = 符合条件订单的支付金额之和（与退款单列）。

### 1.1 线下成交（OfflineDeal 台账）

| 项 | 口径 |
|----|------|
| 事实来源 | 表 `OfflineDeal`（非平台同步订单） |
| 时间字段 | 按 `dealAt`（成交时间）落入统计周期 |
| 计入支付金额 / 总 GMV | `status=confirmed` 且 `amountCent>0` 且未软删 |
| 不计入 | `draft` / `cancelled` / `voided` |
| 主播归属 | **仅人工**；无直播场次 / 排班 / 时段匹配；canonical 类型 `offline_manual` |
| 未指定主播 | 可确认成交并计入 **未归属 GMV**；不得静默归入默认主播 |
| 公式 | 总 GMV = 线上 GMV + 线下 GMV；总 GMV = 已归属主播 GMV 合计 + 未归属 GMV |
| 改归属 | 总 GMV 不变；金额从原主播移除并计入新主播；写审计日志 |
| 退款 | 与线上一致：支付金额仍计全额；`refundCent` 记入退款金额；净额 = amount − refund（≥0） |
| 防重 | `dealKey` / `externalKey` 唯一；禁止用已有平台 P 单号作线下编号 |

**验收**：`npx tsx apps/server/scripts/accept-offline-deal-gmv.ts`

## 2. 支付订单数

| 项 | 口径 |
|----|------|
| 定义 | 统计期内 **有支付时间且已支付** 的订单数 |
| 与支付金额 | 同一套「已支付」判定 |

**验收方法**：支付订单数 = 计入支付金额的订单笔数。

---

## 3. 退款金额

| 项 | 口径 |
|----|------|
| 统计方式 | **单独统计**，不从支付金额中扣减 |
| 时间字段 | 按 **售后成功/退款成功时间**（`refund_ok_time` / `refund_time` 等）落入统计周期 |
| 订单范围 | 仅统计 **支付时间已在统计周期内** 的订单；对这些订单，只合计其退款成功时间落在周期内的有效成功售后金额 |
| 数据来源 | 售后工作台 / 官方接口可确认的有效成功售后 |
| 排除 | 售后中、已撤销、已关闭、拒绝退款、取消售后等非成功状态 |
| 展示 | 与支付金额并列展示，文案用「退款金额」 |

**验收方法**：单日范围下，退款金额应等于该日退款成功时间落入当天的有效成功售后合计；不应把订单支付日之后发生的退款全部回算到支付日。

---

## 4. 品退（商品问题售后）

| 项 | 口径 |
|----|------|
| 定义 | **商品问题 / 质量问题** 相关售后才算品退 |
| 识别 | 需结合：官方品退（品质负反馈）接口、售后原因、成功售后状态 |
| 交叉 | 官方品退为主，售后原因作印证（见 `quality-refund-cross-verify` 等服务） |
| 数据存在性 | 本月实际存在品退数据时，页面 **不能** 显示为 0 |

**验收方法**（涉及品退必做）：

1. 经营总览 — 品退单数 / 品退率（长周期）
2. 主播业绩 — 品退指标
3. 买家排行 — 品退重度客户
4. Drawer — 品退明细列表
5. 明细中 — 售后原因字段是否正确

## 5. 买家唯一标识

| 优先级 | 规则 |
|--------|------|
| 1 | 优先使用 `buyerId` |
| 2 | 缺失时使用 `nick:{昵称}` 兜底 |
| 禁止 | **不能**仅按昵称聚合（同名买家需可区分） |

**验收方法**：抽样的同名昵称是否按 buyerId 分成多条；Drawer 识别码与 buyerKey 一致。

## 6. 多直播号

| 场景 | 规则 |
|------|------|
| 经营总览 / 主播业绩 / 买家排行大盘 | **合并** 所有已启用直播号 |
| Drawer / 主播详情 / 买家详情 | 展示 **来源直播号** |

## 7. 金额格式

| 规则 | 示例 |
|------|------|
| 完整显示，两位小数 | `¥10,079.90` |
| 禁止缩写 | 不得使用 `¥1.01万` |
| 实现 | 前端 `formatMoneyDisplay` 等函数不得 `/10000` 或拼接「万」 |

## 8. 买家排行（全量画像）

| 项 | 口径 |
|----|------|
| 范围 | 历史订单 **累计**，不按经营总览日期过滤 |
| 更新 | 默认每日 03:00 自动重建；可手动重建 |
| 与经营同步 | 相互独立；重建买家排行 **不触发** 30 分钟经营同步 |

## 9. 日期范围与率指标

| 预设 | 签收率 / 品退率 |
|------|-----------------|
| 今日、昨日、本周 | 不强行展示（样本过短易误导） |
| 本月、上月、自定义（足够长） | 适合展示 |

## 11. 日报直播场次与排班归属口径

| 项 | 口径 |
|----|------|
| 直播时段来源 | **仅**来自 Prisma `xhsRawLiveSession`（官方 sellerLiveDetailData 同步入库），按四店官方账号分别读取 |
| 禁止伪造 | **不得**用订单支付时间、排班起止时间、兜底标题生成或冒充直播时段 |
| 排班用途 | `anchorDailySchedule` 当日生效排班（manual → generated_default → virtual_template）用于：**判定哪场真实直播归哪个主播**、**迟到/早退**、**场次标签/店铺展示**；**不**再作为订单支付时间的归属窗口 |
| 直播场次→主播 | 店铺名归一化 + 与排班行重叠分钟数最大者；一场直播只归一个主播 |
| **订单归属（6.13 起）** | 支付时间须落在当日该直播号 **真实开播~下播** 时段内（`xhsRawLiveSession`）；归属到该场次已匹配的主播 |
| 订单回退 | 仅当该直播号当日 **无任何已同步真实场次** 时，才回退用生效排班时段归属 |
| 订单未命中 | 已有真实场次但支付时间落在场次外 → `未归属`（如早开播前、晚下播后的零星单） |
| 未匹配场次 | 进入 `unassignedSessions`，计入 summary 总直播时长，**不计入**任何主播个人时长/流量/直播时段 |
| 去重 | 第一层 `shopCode::liveId` 保留时长更长/字段更完整者；第二层同店同日重叠 ≥30 分钟合并 |
| 无真实直播 | 主播行 `liveTimeRange` 显示「未读取到直播场次」，流量指标为 null（前端 `--`），排班时段仍可单独展示 |
| 调试接口 | `GET /api/board/daily-report/debug-live-sessions?date=YYYY-MM-DD`（只读排查） |

### 11.1 订单归属优先级（6.13 起，上海时区）

| 优先级 | 来源 | 条件 | 结果 |
|--------|------|------|------|
| 1 | 手动指定 | 系统设置里对该 P 订单号有手动归属 | 用手动主播 |
| 2 | **真实直播时段** | 该直播号当日已同步场次，且支付时间 ∈ [真实开播, 真实下播] | 归该场次主播 |
| 3 | 未命中真实时段 | 该直播号当日已有场次，但支付时间在场次外 | **未归属** |
| 4 | 排班回退 | 该直播号当日 **无任何** 已同步真实场次 | 用生效排班表时段归属 |
| 5 | 仍未命中 | — | **未归属** |

**排班表 vs 真实直播的分工**

| 用途 | 用排班 | 用真实直播 |
|------|--------|------------|
| 判定「这场直播是谁播的」 | ✓（与排班重叠最大） | 场次起止来自同步 |
| 订单支付时间归属窗口 | ✗（仅无场次时回退） | ✓ |
| 迟到 / 早退 / 出勤展示 | ✓ | 对比真实开播/下播 |
| 日报展示的直播时段 | — | ✓（`liveTimeRange`） |

**典型样例**

| 场景 | 排班 | 真实直播 | 支付时间 | 归属 |
|------|------|----------|----------|------|
| 早开播 | 09:30–14:00 | 09:25–14:00 | 09:27 | 子杰（在真实时段内） |
| 早开播前下单 | 09:30–14:00 | 09:25–14:00 | 09:20 | **未归属**（早于真实开播） |
| 拖播晚下播 | 09:30–14:00 | 09:28–14:05 | 14:01 | **小红**（在真实 14:05 下播前） |
| 拖播后下单 | 09:30–14:00 | 09:28–14:05 | 14:06 | **未归属**（已下播） |
| 当日未同步场次 | 09:30–14:00 | 无 | 10:00 | 按排班回退归对应主播 |

验收脚本：`npm run verify:live-session-order-attribution`

## 变更流程

1. 改代码前确认是否影响上表口径  
2. 同时检查前端展示与后端 `board-*` / `buyer-ranking-*` 服务  
3. 更新本文件相应章节  
4. 在交付说明中写明验收步骤与样例数据
